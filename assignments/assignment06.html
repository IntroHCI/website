Assignment 6: Meat on the bones

Assignment

List Future Changes from HE Results

Start by taking a look at the feedback you got last week from the Heuristic Evaluations of your prototypes. If you didn't get any feedback at all, either (a) go through the list of heuristics yourself and find any that are violated, or (b) ask a friend for help (we recommend (b): it is easy to overlook your own mistakes). Distill the HE results into a list of concrete changes you want to make to your prototype. Use what worked, and improve what didn't.

Develop a User Evaluation Plan

Next, develop a user test plan. Things that you will want to think about:

  • What are you testing?
  • What do you want to learn? What concerns, questions, and goals is the test focusing on?
  • Where/when will the test take place?
  • What tasks will you ask the user to perform that will help answer your questions?
  • How will your participants share their feedback? (Think aloud, questionnaire/survey, post-test interviews...)
Here is a template you can use.

Complete Interactivity

You should be spending your time fleshing out your interactive prototype. Focus on getting the flow of your interaction working. By the end of this week your prototypes should be fully interactive and ready to be tested.By fully interactive, we mean that all interface elements should be created (and properly linked to each other if applicable) and a user should be able to navigate through your site to perform specific tasks. If you planned it out right and you have been doing your work, you should be in good shape. If not, it's time to step it up. You will get much less out of user testing if you are still struggling with incomplete interactions and a half finished interface.

Update your Development Plan

Continue to update your development plan as you complete tasks and/or revise where necessary. If you haven’t yet, add more tasks for the rest of your project.

What’s this for? A UX agency perspective

by Mike Davison, Community TA and UX Project Manager

‘Where to stop’ with a prototype can be hard to judge. Lets say you have a flight search application.

You will probably want to demonstrate the process of searching, selecting and booking ONE flight, and test that this works. When you present your search results screen, only one flight needs to work to prove your point; the others can be static. Just ensure you highlight that to the user, for example a speech bubble stating ‘book this flight’.

Always ask - is this useful? Do you learn any more from your flight search application if every flight ‘works’? We recently completed a UX design project for a banking software application. To populate every field with data on shares, live information on stock prices, current account balances, and making sure all transfer functions worked would have been incredibly time consuming, so we focussed on one share type, one payment time and importantly, one client type. It told us all we need to know - the rest is a variation on the same theme.

Remember, the point of a prototype is to demonstrate the ‘idea’ of an application, without the need to code the whole thing as it will stand when it’s finished. Just ensure the user can complete the task!

Common Peer Feedback (“I wish…”)

The following statements are common peer feedback given on this assignment. We call these statements "I wish" feedback because they are a way to express things that graders wish the assignment had contained. You could think of these as common problems to avoid.

  • Clarify the concerns, goals, and expectations of the user tests.
  • Make the user tests more structured.
  • Make the user tests more consistent across participants.
  • Make the prototype more interactive so the user test represents a more real-life interaction.
  • Determine the implications of the user succeeding (or not) on each task on the prototype.
  • Make fewer assumptions about users/Reduce bias in user test.
